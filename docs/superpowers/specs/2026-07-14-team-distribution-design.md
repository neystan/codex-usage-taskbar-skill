# Codex Usage Taskbar Team Distribution Design

## Goal

Prepare one private GitHub repository that gives colleagues two supported paths:

1. Install a Codex plugin whose Skill can create or adapt a taskbar quota widget for the colleague's local environment.
2. Copy and build the verified WPF source template, or download a ready-to-run Windows x64 release package.

## Chosen approach

Use one marketplace-backed plugin plus a GitHub Release artifact.

- The repository contains a standard Codex marketplace at `.agents/plugins/marketplace.json` and one plugin named `codex-taskbar-usage-widget`.
- The plugin contains the UTF-8 Skill and a complete source template as plugin assets. The Skill identifies the template as the preferred starting point for changes, while preserving its existing guidance for creating a fresh implementation when a template is not appropriate.
- The repository keeps the canonical WPF solution, its MSTest tests, and the autostart script as normal source-controlled files. A packaging script copies the canonical source into the plugin asset template after validating the solution.
- A release script publishes the widget and launcher as `win-x64`, self-contained binaries, produces a ZIP file, and writes a checksum. The ZIP is prepared locally and is intended for a GitHub Release after the user supplies the repository address. It is not committed to Git.

## Repository layout

```text
.agents/plugins/marketplace.json
plugins/codex-taskbar-usage-widget/
  .codex-plugin/plugin.json
  skills/codex-taskbar-usage-widget/SKILL.md
  assets/source-template/
src/CodexUsageTaskbar/
src/CodexUsageTaskbar.Launcher/
tests/CodexUsageTaskbar.Tests/
scripts/install-autostart.ps1
scripts/sync-plugin-template.ps1
scripts/package-release.ps1
docs/INSTALL.md
README.md
```

`src/`, `tests/`, and `scripts/install-autostart.ps1` remain canonical. `assets/source-template/` is generated by `sync-plugin-template.ps1`, committed as a plugin asset, and must match the source files needed to build, test, and install the widget. Build outputs, `bin/`, `obj/`, and release ZIP files are excluded by `.gitignore`.

## User workflows

### A. Skill-directed implementation

A colleague clones the repository, adds the marketplace, installs the plugin, and asks Codex to create or customize a taskbar usage widget. The Skill instructs Codex to begin from the bundled template when it satisfies the request; otherwise Codex builds a tailored implementation using the Skill's safety, refresh, UI, and lifecycle rules.

### B. Source reuse

A colleague copies the bundled template with the provided script. The script refuses to overwrite a non-empty destination. The copied project contains the solution, widget source, launcher, tests, and autostart installer, and can be built with the documented `dotnet` commands.

### C. Direct use

A colleague downloads the self-contained `win-x64` ZIP from the GitHub Release, extracts it, launches the provided launcher, and can install the current-user scheduled autostart task. No .NET runtime is required. The expected uncompressed footprint is approximately 168-172 MiB, based on the locally installed .NET 9 runtime packs.

## Safety and compatibility

- The widget never copies Codex credentials or modifies Codex binaries.
- The source template targets Windows and uses .NET 9 WPF. The self-contained release targets only `win-x64`.
- The packaging script must stop with an error if the full solution test run fails or the expected application and launcher executables are absent from the publish output.
- The source-copy script must stop with an error when the target directory already exists and contains files.
- The plugin manifest contains only supported plugin metadata; marketplace policy is `AVAILABLE` and authentication policy is `ON_INSTALL`.

## Validation

Before publishing the repository or release:

1. Run `dotnet test CodexUsageTaskbar.sln`.
2. Run the template synchronization script and validate that the asset template contains the solution, source, tests, and installer.
3. Validate the plugin manifest and marketplace structure with the plugin validation script.
4. Run the release packaging script and verify that the ZIP, SHA-256 file, widget executable, and launcher executable exist.
5. Inspect the Git status to ensure build output and release artifacts are ignored while all source, source-template, plugin, and documentation files are tracked.

## Out of scope

- Publishing to GitHub, creating a GitHub Release, or granting repository access until the user provides the repository address and authorizes that external action.
- macOS, Linux, ARM64, or non-Windows packaging.
- Automated updates of already-downloaded release ZIP files.
